# Knowledge Log

- 2026-03-30 — Azure Service Bus DI tests can stay infrastructure-free by registering derived `ServiceBusClient` and `ServiceBusAdministrationClient` instances that use the SDK’s mock-friendly constructors; this is enough to exercise `AddAzureServiceBusSdk()` resolution and missing-client error paths without connection strings or network calls.
- 2026-03-30 — Blob DI tests can stay infrastructure-free by constructing `BlobServiceClient` with a dummy HTTPS account URI plus `AzureSasCredential`; factory-resolution coverage does not require a real storage account or any network I/O.
- 2026-03-30 — The S01 Blob seam intentionally omits explicit container-creation APIs, so the in-memory Blob backend treats `GetContainer("name")` as establishing that container namespace and then keeps blob upload/download state under the shared `InMemoryBlobState` root.
- 2026-03-30 — Key Vault DI tests can stay infrastructure-free by constructing `SecretClient` with a dummy vault URI plus a custom `TokenCredential`; `AddAzureKeyVaultSdk()` resolution coverage does not require a real vault, network I/O, or Azure identity setup.
- 2026-03-30 — Running multiple `dotnet test` commands in parallel against this solution from the same worktree can trigger transient `MSB3026` copy-retry warnings because the projects share `bin/obj` outputs; use a single authoritative `dotnet test ./Azure.InMemory.sln` run when collecting verification evidence.
- 2026-03-30 — In this worktree, shell verification is more reliable with the explicit relative solution path `./Azure.InMemory.sln`; the bare `Azure.InMemory.sln` spelling can be mis-resolved by some command wrappers.
- 2026-03-30 — In-memory Service Bus topic publishes are observable only on canonical subscription entity paths like `<topic>/Subscriptions/<subscription>`; `GetPendingMessages(topicName)` should stay empty after publish, and each subscription receives its own cloned envelope plus independent application-properties dictionary.
- 2026-03-30 — In-memory Service Bus processor handler failures are treated as terminal errored outcomes, not implicit requeues: successful-but-unsettled handlers stay pending, while handler exceptions or invalid second-settlement attempts are recorded in `GetErroredMessages(entityPath)` and surfaced through `ProcessErrorAsync` when configured.
- 2026-03-30 — In-memory Service Bus processors drain a snapshot of the current pending batch synchronously; if a handler returns successfully without settling, the same envelope is requeued only when `AutoCompleteMessages` is false, while `AutoCompleteMessages = true` records the message in `GetCompletedMessages(entityPath)` instead.
- 2026-03-30 — Redelivery bookkeeping is isolated by the canonical entity path, not by topic name: topic publishes clone one envelope per `<topic>/Subscriptions/<subscription>`, and only the subscription path that actually fails should see `DeliveryCount` progression, retry requeues, or max-delivery dead-letter outcomes.
- 2026-03-30 — Key Vault runtime proof now lives in `tests/Azure.InMemory.Tests/KeyVault/InMemory/InMemoryKeyVaultBehaviorTests.cs`; keep `KeyVaultProviderRegistrationTests` limited to DI resolution and diagnostics, because the in-memory secret store behavior contract is now explicitly: missing lookups return `null`, secret names are case-insensitive, and overwriting the same logical secret generates a fresh latest version while shared `InMemoryKeyVaultState` preserves one logical name.
- 2026-03-30 — Blob runtime proof now lives in `tests/Azure.InMemory.Tests/Blob/InMemory/InMemoryBlobBehaviorTests.cs`; keep `BlobProviderRegistrationTests` limited to DI resolution and diagnostics, because the in-memory Blob contract is now explicitly: missing blobs return `false`/`null`, duplicate uploads fail unless `overwrite: true`, container/blob names are case-insensitive, downloads preserve `contentType` while cloning `BinaryData`, and `GetContainer("name")` establishes the in-memory container namespace before upload.
- 2026-03-30 — Across resources, the cleanest proof pattern is: keep provider registration tests DI-only, put runtime behavior in dedicated resource-focused suites, and rely on singleton shared-state roots for low-noise test observability. This preserved traceability in S03-S05 without widening the public seams.
- 2026-03-30 — If a slice delivers truthful behavior earlier than the roadmap originally expected, later slices should prefer tightening proof location and verification over re-implementing the same behavior just to match the original plan wording; preserving the public seam is usually the better milestone outcome.
- 2026-03-30 — In milestone worktrees under `.gsd/worktrees/*`, verify restored solution files resolve under the active `$PWD` before running `dotnet` commands; a sibling worktree such as `../M001` can be the only complete source snapshot, but later execution must target copied files in the active root, not the sibling path.
- 2026-03-30 — Service Bus redelivery applies processor-specific `MaxDeliveryCount` at the canonical entity path that is actually being processed; a failing queue or subscription clone can advance to delivery-count exhaustion and dead-letter independently while untouched sibling subscription clones keep their own pending copy and default delivery metadata until their own processor runs.
- 2026-03-31 — In this WSL worktree, the package-inspection verification should use `python3`, not `python`; the `python` alias is absent even though the slice plan’s inline command uses `python -c`.
- 2026-03-31 — `InMemoryServiceBusState` is public, but its `GetSubscriptionEntityPath` helper is internal; package-facing docs and external consumers must inspect subscription outcomes with the literal canonical entity-path string `<topic>/Subscriptions/<subscription>`.
- 2026-03-31 — In this M002 worktree, foreground `bash` resolved `./Azure.InMemory.sln` correctly for `dotnet test`, but `async_bash` reported `MSB1009` against the same relative path; prefer foreground `bash` for path-sensitive dotnet verification here.
